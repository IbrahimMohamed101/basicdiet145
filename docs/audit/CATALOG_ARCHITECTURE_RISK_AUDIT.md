# Catalog Architecture Risk Audit

Date: 2026-06-04

## Summary

This audit traces catalog correctness risks across the canonical menu seed, public one-time menu serializer, subscription planner catalog, planner validation, publishing, and documentation. The main architecture shape is sound: `Menu*` collections and relation collections are the canonical catalog, while `Builder*`, `Sandwich`, and `SaladIngredient` remain compatibility mirrors.

The highest remaining risks are duplicated seed/runtime allowlists, compatibility mirror drift, and sync/publish behavior that can silently overwrite or preserve the wrong catalog rows depending on mode.

Resolved note: the subscription `premium_large_salad` protein policy has been clarified. It allows only `SUBSCRIPTION_PREMIUM_LARGE_SALAD_PROTEIN_KEYS` and rejects premium proteins plus `extra_protein_50g`.

Resolved note: duplicated high-risk allowlists are now protected by static parity tests in `tests/catalogAllowlistParity.test.js`. This does not fully remove duplication, but it prevents silent drift for premium-large-salad proteins/excluded groups, standard proteins, premium meal proteins, canonical carbs, and subscription sandwich keys.

## Coverage Gaps To Add

- Seed bootstrap contract test for current `scripts/seed-catalog.js`, separate from older `seed-one-time-menu` expectations.
- Parity test that `scripts/seed-catalog.js` premium-large-salad protein allowlist equals `SUBSCRIPTION_PREMIUM_LARGE_SALAD_PROTEIN_KEYS`.
- Public menu test proving `basic_meal` allowed proteins come from product relations or a shared exported constant, not a stale serializer hard-code.
- Planner API test proving `builderCatalogV2.premium_large_salad` omits `extra_protein_50g` and validation rejects submitted extra-protein selections.
- Dashboard edit test for mirror drift: edit canonical protein/carb/sandwich fields and verify planner catalog/validation reads the updated intended source.
- Publish test that simulates public snapshot failure and asserts the chosen behavior: block publish or persist an explicit snapshot error.

## Recommended Next Fix Order

1. Add current `seed-catalog.js` contract tests, then update stale catalog docs.
2. Decide mirror ownership and add dashboard-to-planner regression tests.
3. Harden publish/sync behavior with explicit failure reporting and stale-relation audits.
4. Optionally extract seed allowlists into a pure data module once broader seed refactoring is in scope.
